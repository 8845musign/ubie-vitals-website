## Introduction

- 覆い被さる画面に対して、何かしらのアクションを実行させるために使用します
  - 「シェア」「削除」「通報」など
- 通常のモーダルダイアログとは違い、コンテンツの中央を占有しないため画面のコンテキストを維持しやすい特徴があります

## Usage

- 極力1つのモーダルにつき、アクションは1つにとどめましょう
- モバイル環境に最適化されています
  - モバイルでは、ユーザーが操作しやすい領域は画面下部になります
  - PCユーザーにとっては、逆に操作しづらい可能性があります。画面下部はポインターを置くことが少ないため
- 領域をアクションで占有できるため、タップエリアを十分確保できます
- PCやタブレットなどの、広い画面においては使用してはいけません
  - 前述の通り、操作しづらい可能性があります
  - 不自然に領域を占めてしまます
  - ドロップダウンメニューなどの、大体UIを検討してください
- 複雑な操作をさせてはいけません
  - ページとして分割することを検討してください
  - モーダルダイアログはURLを持たないため[^modal]、何かの拍子にページを離脱してしまうと操作内容が失われます
- Modal内部にコンテンツが多く、画面領域の大部分を占めてしまうような使い方はしてはいけません
  - 通常のModalを利用したり、ページを分割したりすることを検討してください
  - どうしても `<ActionHalfModal>` での実装をしなくてはならない場合、内部のコンテンツをスクロールすることを検討してください
- 元の画面と全く関係のないコンテンツを表示するのは避けてください
- 「キャンセル」「閉じる」ボタンの文言を使い分けてください
  - 1つのアクションや導線にユーザーを誘導した場合は「キャンセル」
    - 「キャンセル」は、すでに確立しつつあるフローやアクションを中断するニュアンスがあります
  - 複数のアクションをユーザーに提示する場合は「閉じる」
    - より汎用的な意味合いがあります
    - 「閉じる」が配置されたダイアログを開いた場合、ユーザーは自身が実行したいアクションを決めていない状態であるかもしれません
    - ただ、情報が提示されるモーダルダイアログの場合「閉じる」の方が適切です

[^modal]: 厳密にはモーダルダイアログはURLを持つこともできますが、実装コストが高くなります。また、場合によっては不自然なUIとなる可能性があります。モーダルダイアログは基盤となる画面とセットで設計されることが多いためです。

## Accessibility

- `header` propに渡されたテキストは、`aria-labelledby` 属性を通してダイアログ自体を説明するラベルとなります[^aria-labelledby]。
  - 特にスクリーンリーダーユーザーにとって意味のあるものとなります

[^aria-labelledby]: [ARIA: dialog ロール - アクセシビリティ | MDN](https://developer.mozilla.org/ja/docs/Web/Accessibility/ARIA/Roles/dialog_role)
